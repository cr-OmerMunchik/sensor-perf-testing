System CPU - Average by Role

What this shows: Total system CPU utilization (all processes combined) averaged over each scenario. Sourced from Windows win_cpu _Total counter, already normalized to 0–100%.

Why baseline (S1) can show higher CPU than sensor hosts: This metric measures CPU used by all processes, including the test workload scripts themselves (PowerShell, cmd, etc.). The sensor intercepts system calls (file I/O, process creation, registry operations), which throttles the workload, causing it to run longer at lower average CPU intensity. This does not mean the sensor reduces CPU usage — it means the same work is spread over more wall-clock time. The true sensor CPU overhead is isolated in the “Sensor CPU” and “Process CPU Impact” sections below.

Bottom Line

Memory Impact — V26.1 + Phoenix:
• Sensor process memory (Working Set): V26.1+Phoenix sensor processes consume an average of 187 MB across all scenarios, peaking at 908 MB during file_stress_loop.
• Comparison across versions (sensor Working Set): V26.1+Phoenix: avg 187 MB / peak 908 MB | V26.1+Legacy: avg 556 MB / peak 1111 MB (file_storm) | V24.1+Legacy: avg 594 MB / peak 858 MB (network_burst).
• System-level memory overhead vs. baseline: Total OS-reported available memory delta (S2 minus S1) shows avg -47 MB, peak 76 MB during user_account_modify. Note: This metric is noisy — it reflects the net effect of the sensor, OS file cache, and workload differences across VMs. Negative values mean the sensor host had more free memory than the baseline, typically because the workload itself consumed more OS cache on the baseline VM. The sensor Working Set numbers above are a more reliable measure of sensor memory consumption.
CPU Impact — V26.1 + Phoenix:
• Sensor processes average 7.3% CPU across all scenarios, peaking at 31.5% during service_cycle.
• Comparison across versions (avg sensor CPU): V26.1+Phoenix: 7.3% | V26.1+Legacy: 2.7% | V24.1+Legacy: 4%. V26.1+Phoenix uses 82% more CPU than V24.1+Legacy.
Heaviest Process on V26.1 + Phoenix: ActiveConsole during browser_streaming averaged 24% CPU (peak: 24.6%).
For comparison — Nnx Process (V24.1 Only): Nnx is a file-monitoring component present only in sensor V24.1 (removed in V26.1). During file_stress_loop, Nnx averaged 22.1% CPU (peak: 28.2%). Nnx reacts heavily to file I/O and can continue processing its backlog long after the workload ends. V26.1 eliminates this bottleneck entirely.

Conclusions

  • CPU: V26.1+Phoenix sensor CPU peaks above 15% in: service_cycle (31.5%), file_stress_loop (27%), browser_streaming (25.7%).
  • Memory: V26.1+Phoenix sensor process Working Set: avg 187 MB, peak 908 MB. Comparison: V26.1+Legacy avg 556 MB / peak 1111 MB, V24.1+Legacy avg 594 MB / peak 858 MB — Phoenix has the lowest sensor memory footprint.
  • V24.1 Sensor (Nnx): The Nnx component on V24.1 consumes significant CPU during file-heavy workloads and continues processing after workloads end. V26.1 eliminates this bottleneck entirely.
  • Backend: Legacy backend shows 4.6% lower sensor CPU than Phoenix.
  • Test Coverage: 13 out of 13 scenarios completed across sensor hosts. All scenarios completed successfully on S2 and S3.
  • Disk I/O: Largest write throughput difference between sensor and baseline hosts observed during file_storm.
