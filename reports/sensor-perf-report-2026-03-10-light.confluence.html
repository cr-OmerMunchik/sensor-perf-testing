System CPU - Average by Role

What this shows: Total system CPU utilization (all processes combined) averaged over each scenario. Sourced from Windows win_cpu _Total counter, already normalized to 0–100%.

Why baseline (S1) can show higher CPU than sensor hosts: This metric measures CPU used by all processes, including the test workload scripts themselves (PowerShell, cmd, etc.). The sensor intercepts system calls (file I/O, process creation, registry operations), which throttles the workload, causing it to run longer at lower average CPU intensity. This does not mean the sensor reduces CPU usage — it means the same work is spread over more wall-clock time. The true sensor CPU overhead is isolated in the “Sensor CPU” and “Process CPU Impact” sections below.

Bottom Line

Memory Impact — V26.1 + Phoenix:
• Sensor process memory (Working Set): V26.1+Phoenix sensor processes consume an average of 654 MB across all scenarios, peaking at 1263 MB during service_cycle.
• Comparison across versions (sensor Working Set): V26.1+Phoenix: avg 654 MB / peak 1263 MB | V26.1+Legacy: avg 853 MB / peak 1525 MB (service_cycle) | V24.1+Legacy: avg 689 MB / peak 759 MB (user_account_modify).
• System-level memory overhead vs. baseline: Total OS-reported available memory delta (S2 minus S1) shows avg 613 MB, peak 895 MB during idle_baseline. Note: This metric is noisy — it reflects the net effect of the sensor, OS file cache, and workload differences across VMs. Negative values mean the sensor host had more free memory than the baseline, typically because the workload itself consumed more OS cache on the baseline VM. The sensor Working Set numbers above are a more reliable measure of sensor memory consumption.
CPU Impact — V26.1 + Phoenix:
• Sensor processes average 14.5% CPU across all scenarios, peaking at 32.7% during file_stress_loop.
• Comparison across versions (avg sensor CPU): V26.1+Phoenix: 14.5% | V26.1+Legacy: 14.6% | V24.1+Legacy: 2.5%. V26.1+Phoenix uses 480% more CPU than V24.1+Legacy.
Heaviest Process on V26.1 + Phoenix: ActiveConsole during driver_load averaged 15.8% CPU (peak: 23.1%).
Nnx Process Impact: Nnx is a file-monitoring component present in all sensor versions. During combined_high_density, Nnx averaged 8.6% CPU (peak: 17.7%). Nnx reacts heavily to file I/O and can continue processing its backlog long after the workload ends, consuming significant CPU.

Conclusions

  • CPU: V26.1+Phoenix sensor CPU peaks above 15% in: file_stress_loop (32.7%), combined_high_density (25.8%), service_cycle (25.7%).
  • Memory: V26.1+Phoenix sensor process Working Set: avg 654 MB, peak 1263 MB. Comparison: V26.1+Legacy avg 853 MB / peak 1525 MB, V24.1+Legacy avg 689 MB / peak 759 MB — Phoenix has the lowest sensor memory footprint.
  • Nnx Component: The Nnx file-monitoring component (present in all versions) consumes significant CPU during file-heavy workloads and can continue processing its backlog after workloads end.
  • Backend: No significant CPU difference between Phoenix and Legacy backends.
  • Test Coverage: 13 out of 13 scenarios completed across sensor hosts. All scenarios completed successfully.
  • Disk I/O: Largest write throughput difference between sensor and baseline hosts observed during combined_high_density.
